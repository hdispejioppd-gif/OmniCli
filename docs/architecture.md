# Архитектура OmniCLI

## Обзор

OmniCLI — provider-neutral runtime для агентных CLI-приложений. Основные слои:

```
CLI / TUI / Workflow / Supervisor
            │
      Agent (оркестрация, верификация, цикл)
            │
    Provider (fake, openai, anthropic, ollama)
            │
    Tools (файлы, shell, git, плагины, MCP)
            │
Permission + Store (SQLite) + Events (NDJSON)
```

## Модули

| Модуль | Назначение |
|--------|-----------|
| `agent` | Provider-neutral цикл оркестрации и верификации |
| `provider` | Абстракция модели + адаптеры fake/OpenAI/Anthropic/Ollama |
| `tools` | Ограниченное выполнение процессов, атомарные файловые операции, Git, shell, плагины |
| `plugin` | Хост подпроцессных плагинов через JSON-RPC over stdio |
| `permission` | Централизованная политика capability |
| `store` | SQLite-сессии, сообщения, журнал событий |
| `events` | Версионированный протокол исполнения |
| `config` | Типизированная конфигурация и дефолты проекта |

## Ключевые ограничения безопасности

- Запись и shell требуют явных флагов (`--allow-write`, `--allow-shell`).
- `apply_patch` требует SHA-256 файла и exact-match текста.
- `create_file` не перезаписывает существующие файлы.
- MCP, плагины, worktree — отдельные capability-флаги.

## Встроенные инструменты

- Навигация и чтение: `list_directory`, `read_file`, `repo_map`, `search_code`, `search_files`.
- Правки файлов: `create_file`, `apply_patch`, `move_file`, `delete_file`, `copy_file`, `create_directory`, `delete_directory` (рекурсивно, без удаления корня), `replace_in_files` (поиск-замена по всему проекту, литеральная или через регулярные выражения с `regex=true` и группами `$1`; `dry_run=true` — предпросмотр без записи).
- Git: `git_status`, `git_diff` (read-only) и `git_commit` (стейджит изменения и делает коммит, `--no-gpg-sign`, требует `--allow-write`).
- LSP-навигация (реальный LSP-клиент поверх `src/lsp.rs`; rust-analyzer / pyright / typescript-language-server / gopls по умолчанию, либо `server`-переопределение; координаты 1-based):
  - `goto_definition`, `hover`, `find_references`, `workspace_symbols` — чтение, запуск сервера требует `--allow-shell`.
  - `rename_symbol` — проектное переименование через `textDocument/rename`; применяет `WorkspaceEdit` только к файлам внутри рабочей папки, есть `dry_run`, требует `--allow-write`.
- Планирование: `update_plan` — короткий упорядоченный план шагов со статусами (pending/in_progress/completed), как у Codex и Claude Code; без побочных эффектов.
- Проверка: `run_checks` (Rust/Node/Python, ограниченный argv-запуск). Разбирает вывод в структурированный `failures[]` (файл/строка/колонка/сообщение): `cargo`-ошибки через `error/warning: ...` + `--> file:line:col`, `tsc`/generic для Node, `FAILED/ERROR ...` для pytest. Поддерживает фокусный перезапуск: `filter` (имя теста → `cargo test <f>` / `pytest -k <f>` / `npm run <s> -- <f>`) и `ecosystem` (rust|node|python). Это основа цикла авто-починки: запустить → прочитать `failures[]` → исправить → перезапустить только упавший тест.
- Хост: `shell` (Windows cmd/PowerShell + Unix sh) под флагом `--allow-shell`.
- Интернет: `web_search`, `web_fetch`, `web_download` (сохранение файла из URL в рабочую папку).

## Экосистема: MCP и плагины

- Подключённые MCP-серверы (блоки `[mcp.servers.<name>]` в `omni.toml`) автоматически подключаются при старте (под `--allow-mcp-start`/`--allow-mcp-call`), и каждый их инструмент проксируется агенту под именем `mcp__<server>__<tool>`.
- Плагины (подпроцессы JSON-RPC over stdio) регистрируются под `plugin__<name>__<tool>` (под `--allow-plugins`).
- Системный промпт агента объясняет эти префиксы, чтобы модель предпочитала выделенные интеграции вместо shell.
- CLI-разведка: `omni mcp list` подключается к настроенным MCP-серверам и печатает их инструменты (есть `--json`); `omni plugins list` — аналогично для плагинов. Отдавать свои инструменты наружу можно через `omni mcp serve`.

## Автоматический fix-цикл

Агент умеет сам чинить проект до «зелёного» состояния. Механизм: при `verify: true` после того, как модель завершает ход без вызовов инструментов и в рабочей папке были изменения, `Agent::run` автоматически запускает `run_checks`. Если проверки падают, структурированный результат (`failures[]` + `failure_count`) возвращается модели как обратная связь с номером попытки и числом оставшихся ошибок, после чего цикл повторяется в пределах `max_turns`. Каждая итерация эмитит `[auto-fix] (fix attempt N; K failing check(s) remaining)` в поток событий.

Отдельная команда `omni fix` — первоклассная точка входа в этот цикл: она подразумевает `--allow-write` и `--verify`, формирует целевой промпт «сделай проверки зелёными», и опционально сужает область через `--ecosystem rust|node|python` и `--filter <name>` (прокидываются в `run_checks` для фокусного перезапуска). Дополнительные подсказки можно передать через `--prompt`.

## Предпросмотр диффов перед записью

Перед тем как запись будет авторизована, агент показывает её дифф. Механизм: у трейта `Tool` есть необязательный метод `preview(arguments, context) -> Option<WritePreview>` (по умолчанию `None`). Он строго только читает файлы и никогда не влияет на саму запись. Превью покрывает весь набор пишущих инструментов (файлы и директории):

- `create_file` — весь новый файл как добавление (`+N строк`).
- `apply_patch` — точный дифф old→new по текущему содержимому файла (только при единственном совпадении `old_text`).
- `delete_file` — всё содержимое файла как удаление (`-N строк`); бинарные помечаются отдельно.
- `move_file` — короткая сводка переименования `− from` / `+ to`.
- `copy_file` — содержимое источника как добавление в новом файле (`+N строк`).
- `create_directory` — какие сегменты пути будут созданы (`+ path/`), учитывая уже существующие родительские каталоги.
- `delete_directory` — read-only рекурсивный обход: файлы и подкаталоги к удалению (`- entry`), до 100 записей, остальные в счётчик; корень рабочего каталога не превьюится.
- `replace_in_files` — read-only обход, зеркалящий логику `execute` (те же исключения и лимиты, литерал/regex): пофайловые диффы до 20 файлов, остальные в счётчик.

`Agent::run` вызывает `preview` сразу после `tool_call_requested` и до авторизации, эмитя событие `diff_preview { path, summary, diff }`. Его отображают и TUI (блок `DIFF ...`), и консольный вывод, и NDJSON-поток. Так пользователь видит, что именно изменится, и — когда запись требует подтверждения (без `--allow-write`) — принимает решение уже с диффом перед глазами. Сам дифф строится построчно через LCS и ограничен (до 400 строк вывода; большие файлы сворачиваются в сводку).

## Поток событий

Каждый запрос получает UUIDv7 и журналируется в SQLite инкрементально. NDJSON-поток версионируется.
